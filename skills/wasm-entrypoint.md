# Sol Wasm Entrypoint Design (mbtx + Mars WASI/WAGI)

## Purpose

Add a mode to `sol` that "executes individually prebuilt wasm per route/feature as an entrypoint".

Target runtimes:

- WASI CLI
- Spin WAGI (`executor = "wagi"`)
- Wasm Component (future)

This design starts from `.mbtx`, but assumes connection with the WASI/WAGI compatibility layer on the `mars` side (`spin_wagi` / `wasm_component`).

## Constraints (as of 2026-02-18)

`.mbtx` is a nightly-only feature with the following constraints:

- Requires `moon >= 0.1.20260214`
- Assumes `fn main` (main package)
- `moon test file.mbtx` is not practical
- Direct specification of custom wasm exports is not possible (`moon.pkg link` cannot be inlined)
- Cross-file imports are not possible (single file only)

Therefore, it is difficult to treat a standalone `.mbtx` as a "wasm API with arbitrary exports".
This feature adopts the "`main`-execution-type wasm entrypoint" approach.

## Scope

### In scope

- Individual prebuild of multiple `.mbtx` files
- Generation of the route-to-wasm artifact mapping table
- Unification of the WAGI response protocol (returning HTTP responses via stdout)
- Generation of Spin manifest fragments

### Out of scope

- Export extensions to `.mbtx` itself
- `.mbtx` support for `moon test`
- Immediate wasm support for all `mars` features (incremental migration)

## High-level Architecture

1. `sol` reads the `wasm-entrypoints` definition
2. Each entrypoint is built independently (prebuild)
3. Outputs the artifact-to-route mapping manifest
4. At runtime, the host (Spin/WASI launcher) starts the corresponding wasm based on the route

```
route request
  -> host router (Spin/WAGI or launcher)
  -> selected prebuilt wasm entrypoint
  -> stdout HTTP response
```

## Configuration Draft

Add `wasmEntryPoints` to `sol.config.ts`.

```ts
export default {
  wasmEntryPoints: [
    {
      id: "users_show",
      route: "/users/:id",
      source: "app/entries/users_show.mbtx",
      runtime: "wagi", // "wagi" | "wasi-cli" | "component"
      method: ["GET"],
    },
  ],
}
```

## Build Pipeline Draft

Each `.mbtx` is built independently, and artifacts are placed at fixed paths:

```bash
moon run app/entries/users_show.mbtx --target wasm-gc --release --build-only
```

Output location (example):

```
.sol/wasm/
  entries/
    users_show/
      module.wasm
      metadata.json
  manifest.json
  spin.toml.fragment
```

`manifest.json` example:

```json
{
  "entries": [
    {
      "id": "users_show",
      "route": "/users/:id",
      "method": ["GET"],
      "runtime": "wagi",
      "wasm": ".sol/wasm/entries/users_show/module.wasm"
    }
  ]
}
```

## Runtime Contract

`.mbtx` entrypoints follow this minimal contract:

- Input:
  - Receives path/method/query/body via env/argv/stdin
- Output:
  - Writes `Status-Line + Headers + Body` to stdout (WAGI compatible)
- Exit code:
  - Normal: `0`
  - Abnormal: `!= 0`

This contract is fixed through templating generated by `sol`.

## Mars Integration Strategy

In the short term, reuse the pure parts of `mars` (router/trie) and absorb the I/O boundary on the `sol` side.

In the medium term, connect with `mars` adapters:

- Generate configuration information for `@mars/adapters/spin_wagi`
- Generate binding hints for `@mars/adapters/wasm_component`

Since the `mars` side also contains placeholder-like implementations for adapters, `sol` takes the lead responsibility of "preparing the artifacts and manifest/hints".

## Proposed Commands

- `sol build --wasm-entrypoints`
- `sol build --wasm-entrypoints --runtime wagi`
- `sol build --wasm-entrypoints --emit-spin-fragment`

Future:

- `sol dev --wasm-entrypoints` (incremental rebuild of only changed `.mbtx` files)

## Migration Plan

### Phase 0 (Doc + Prototype)

- Support for only 1 entrypoint
- Output `manifest.json` and `spin.toml.fragment`

### Phase 1 (Multi-entrypoint)

- Prebuild of multiple entrypoints
- Route conflict detection

### Phase 2 (Mars adapter bridge)

- Formalize `spin_wagi` / `wasm_component` configuration output

### Phase 3 (Operational hardening)

- Build cache
- Error format standardization
- Tracing/logging conventions

## Risks

- `.mbtx` depends on nightly builds
- There are differences in method/body retrieval across WAGI runtimes
- Build/startup costs increase as the number of entrypoints grows

## Decision Record

- Adopted the "per-entrypoint executable wasm" model based on the constraints of `.mbtx` (main-only, no custom exports)
- Unified on the protocol (stdin/stdout/env) model rather than the API export model
- `sol` handles the build orchestrator and manifest generator roles, confining runtime implementation dependencies to the adapter boundary
